## B. 수정 제안 (우선순위순)

### B-1. [높음] `robotgo-hongdae` 크래시 — 근본 원인: 필드명 불일치

**원인** (증상 아님): 렌더 경로 전체가 `trip.stops` + `{name,time,memo}` 한 곳으로 흐르는데(`js/viewer.js:108,114,134`, `js/map.js:17,33-37`), 데이터 파일만 `places` + `{address,url}`. 한 지점만 어긋나서 크래시.

두 가지 결이 있다. **codex는 데이터·CSS까지 넣어 "주소+네이버 링크 표시"를 하려던 것** — 절반만 완성된 기능이다. 그래서 완성이냐 폐기냐를 고른다.

---

#### 안 (A) — **폐기(가장 게으름)**: 데이터만 규격에 맞추고 데드 CSS 삭제

`data/trips/robotgo-hongdae.json`에서 `"places"` → `"stops"` 한 단어만 바꾸면 크래시 사라짐(이름·시간 렌더됨). 대신 `address`/`url`은 버려지므로 §A의 CSS 22줄도 같이 삭제.

- 변경: JSON 키 1개 + `viewer.css` 22줄 삭제.
- 잃는 것: 주소 텍스트·네이버 지도 링크(데이터엔 있는데 안 보임).
- 언제 이걸: 홍대 일정에 주소/링크가 필요 없을 때.

---

#### 안 (B) — **완성(권장)**: 반쯤 만든 기능을 배선. 이미 넣은 데이터·CSS를 살림

codex가 이미 `address`·`url` 데이터와 `.address`·`.map-link` CSS를 넣어놨다. 뷰어 JS만 배선하면 데드 CSS가 살고, 파일 안 정보가 그대로 보인다.

**1) 필드명 정규화** — 파일이 한 곳뿐이니 데이터를 고치는 게 가장 게으름:
`data/trips/robotgo-hongdae.json` `"places"` → `"stops"`.

> 대안(코드측, 비권장): `js/data.js`의 `getTrip`에서 `t.stops = t.stops || t.places` 1줄로 양쪽 수용. 미래 codex 파일까지 커버하지만 코드 경로가 늘어남. `places` 파일이 하나뿐인 지금은 데이터 수정이 더 게으름. `ponytail:` — 두 번째 `places` 파일이 나오면 그때 이 1줄 추가.

**2) 뷰어 카드에 주소·링크 렌더** — `js/viewer.js:117-123` 스톱 카드 `innerHTML`에 두 줄 추가 (필드 있을 때만):
```js
        <div class="name">${esc(s.name)}</div>
        ${s.address ? `<div class="address">${esc(s.address)}</div>` : ""}
        ${s.url ? `<div class="map-link"><a href="${esc(s.url)}" target="_blank" rel="noopener">지도에서 보기 →</a></div>` : ""}
        ${s.memo ? `<div class="memo">${esc(s.memo)}</div>` : ""}
```
→ 이미 있는 `.address`/`.map-link` CSS가 이 순간부터 유효. §A의 -22줄 지적 해소.

- (선택) 지도 팝업에도 링크를 넣으려면 `js/map.js:33-37` 팝업에 같은 패턴 추가. 안 넣어도 카드에서 링크 열림 → **YAGNI, 생략 권장.**
- 변경: JSON 키 1개 + `viewer.js` 2줄. **의존성 0, 신규 파일 0.**
- `esc()`로 이미 이스케이프하므로 XSS 안전. `url`은 데이터 편집자가 신뢰원(네이버 단축링크)만 넣는 전제.

**권장: 안 (B).** 데이터·CSS가 이미 그 기능을 향해 절반 와 있어서, 지우는 것보다 2줄 배선이 낭비가 적다.
